# Plan 0001: General Unit System

## Goal

Add a general, extensible unit system to `lifocalc` that supports:

- quantities like `1[Mb/s]`
- explicit conversion with `in`, e.g. `1[Mb/s] [kb/s] in`
- implicit conversion shorthand, e.g. `1[Mb/s] [kb/s] 2 *`
- derived units through `*` and `/`
- readable auto-formatting with preferred display range `[1, 1000)`

The design should keep the code easy to read and easy to extend.

## Product Decisions

### Runtime semantics

- Store all quantities internally in canonical base units.
- Conversion changes display preference only, not stored magnitude.
- Support:
  - `+` and `-` only for dimension-compatible quantities
  - `*` and `/` for derived units
  - unit cancellation when dimensions cancel
- Support explicit conversion operator:
  - `1[Mb/s] [kb/s] in`
- Support implicit conversion shorthand:
  - bare unit spec `[kb/s]` behaves like `in` on the top compatible quantity
  - `1[Mb/s] [kb/s] 2 *` is equivalent to `1[Mb/s] [kb/s] in 2 *`
  - `2 [kb/s]` is invalid

### Formatting semantics

- Auto-pick a readable display unit whose scaled magnitude is in `[1, 1000)`, when possible.
- For compound units, scale only the first factor during auto-formatting.
- Preserve an explicit display-unit override across later arithmetic when still compatible.
- If explicit display override becomes incompatible after an operation, drop it and fall back to auto-formatting.

### Initial unit scope

- General unit framework
- All SI prefixes (`k`, `M`, `G`, etc.)
- Initial dimensions:
  - data: `b`, `B`
  - time: `s`, `min`, `hour`, `day`
- Specific unit-related errors for:
  - unknown unit
  - incompatible units
  - invalid conversion target
  - invalid implicit conversion usage

## Data Model

Replace the current number-only value model with a quantity model.

### New runtime value

```rust
struct Value {
    magnitude: Magnitude,
    dims: BTreeMap<BaseDimension, i32>,
    display_unit: Option<UnitExpr>,
}

enum Magnitude {
    Exact(BigRational),
    Approx(f64),
}
```

### Unit representation

Use a canonical algebraic representation for units.

```rust
struct UnitExpr {
    dims: BTreeMap<BaseDimension, i32>,
}
```

Keep named-unit lookup separate from runtime values.

```rust
struct UnitDef {
    symbol: &'static str,
    factor: BigRational,
    dims: BTreeMap<BaseDimension, i32>,
}
```

Notes:
- `factor` is used only during parsing/formatting.
- Runtime `Value` stores magnitude already converted into base units.
- `display_unit` is a formatting preference, not part of arithmetic semantics.

## Phase 2: Add unit definitions and parsing primitives

### Files
- new module: `src/engine/units.rs`
- `src/engine/mod.rs`
- `src/engine/parse.rs`

### Work

  - `BaseDimension`

  - known unit registry
- Parse:
  - quantity literals like `1[kB]`
  - bare unit specs like `[kb/s]`
  - unit expressions with multiplication, division, and exponents if needed
- Convert parsed quantities immediately into canonical base units.
- Keep parsing code readable by separating:
  - numeric literal parsing
  - unit expression parsing
  - quantity assembly

### Expected result
- Parser can distinguish:
  - quantity literals
  - unit-spec tokens
  - operators/functions

## Phase 3: Extend evaluator token model

### Files
- `src/engine/eval.rs`
- `src/engine/functions.rs`
- `src/engine/errors.rs`

### Work
- Introduce token/value handling for:
  - quantity values

  - convert the top compatible quantity immediately
  - error if no valid target quantity is available
- Update `+`, `-`, `*`, `/` to use dimension-aware arithmetic.

### Expected result
- Expressions like:

## Recommended Module Layout

To keep the code readable:

- `src/engine/number.rs`
  - runtime `Value` / `Magnitude`
- `src/engine/units.rs`
  - unit registry, prefixes, parsing helpers, formatting helpers
- `src/engine/parse.rs`
  - token classification and literal parsing
- `src/engine/eval.rs`
  - expression/token evaluation
- `src/engine/functions.rs`
  - arithmetic/operator dispatch
- `src/engine/errors.rs`
  - unit-specific error types

## Risks

- Derived-unit formatting can become complex if formatting rules are too clever.
- Implicit conversion must remain predictable and should only act where semantics are unambiguous.
- Preserving explicit display overrides across arithmetic needs clear compatibility rules.
- SI prefix handling plus case-sensitive `b`/`B` requires careful tests.

## Recommended Delivery Order

1. Introduce `Value` while preserving current number-only behavior.
2. Add unit registry and quantity parsing.
3. Add unit-aware arithmetic and explicit `in`.
4. Add implicit conversion shorthand.
5. Add auto-formatting.
6. Add config flag.
7. Add docs and full tests.

## Definition of Done

The feature is complete when:

- quantities with units parse successfully
- mixed compatible units evaluate correctly
- derived units work with `*` and `/`
- `in` works
- implicit conversion shorthand works behind config
- stack/hint formatting shows readable units
- invalid unit operations return specific errors
- unit tests and YAML cases cover the new behavior
- README documents syntax and semantics
